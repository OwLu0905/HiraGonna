
## Requirement:
建立一個日本平假名練習網頁，需求如下：

- 開始練習的按鈕,按下去後出現字,並且需求如下:
    - 有輸入欄可以輸羅馬拼音，按了送出後,先顯示答案的按鈕，並且不要跳下一題，使用者接著可以按下一步換下一題。
    - 平假50音總共有46個，所以一次練習就是46個為一組練習。每次請隨機出現。
    - 請顯示目前回答了幾個
    - 請計算回答時間（注意：送出後顯示答案一直但下一題這區間不要計算!你要計算的是顯示題目到送出的這段時間)
    - 作答完畢時, 顯示總表, 並且每個單字有像是heatmap一樣的顏色,表示回答時間(紅色表示時間較久,至於標準請你先想辦法設定一個紅黃綠標準)
    - 有結束session的按鈕
    - 目前先不用存db, 所以你可能需要自己存狀態,如果需要global可以用zustand
    - 另外日本字體有分,明朝体（Mincho）和教科書体（Kyōkasho-tai), 我認為可以隨機顯示字體, 你覺得是好點子的話就這樣實作

    - NOTE: 看完需求後,請思考還需要哪些功能

- 有一個頁面是看所有的平假名(母音跟子音當作x,y label)
   - 有切換明朝體與教科書體的按鈕, 並標注現在是哪個字體

- 側欄menu目前就先只顯示是Hiragana, 
    - 目前練習都是先練正常的平假名, 之後在開始練習的頁面, 會新增 voiced sounds以及 contracted sounds等, 目前先不動工
    - 之後側欄還會有片假名, 實作一樣的功能, 目前先不動工

- 請邊做UIUX,一邊寫測試,使用vitest, 如果需要使用playwright測試也可以, 測試範圍你先自己抓

## 手機版練習需求（2026-07-20 討論定案，同日實作完成）

背景：手機上打字有兩個問題——(1) 每題送出後 input unmount、focus 跳到「下一題」，OS 鍵盤反覆收合／彈出，版面跳動且污染作答計時（計時從顯示題目開始，但鍵盤還在動畫中無法輸入）；(2) 使用者可能用日文 flick 鍵盤而不是英文鍵盤，輸入的是假名不是羅馬拼音。

- 練習分兩種作答模式，在開始畫面選擇（預設輸入模式）：
    - 輸入模式（現行）：看字輸入羅馬拼音。
    - 選擇模式：出 4~6 個羅馬拼音選項按鈕（隨機排列，一個正確），點選即送出。完全不需要鍵盤、每題計時基準一致，適合手機。干擾選項優先取形近／音近的假名讀音會更有練習效果。

- 輸入模式的手機修正（bug 級，優先做）：
    - 單一常駐 input：送出後 input 不 unmount、focus 不離開；顯示答案狀態下再按一次 Enter 直接進下一題。鍵盤整個 session 不收合。
    - input 屬性：`enterKeyHint`、autocomplete／autocorrect／autocapitalize 關閉；字級維持 ≥16px 避免 iOS 聚焦縮放（目前 text-lg 已符合）。

- 答案判定同時接受羅馬拼音與假名（支援 flick 鍵盤）：
    - flick 打假名同樣證明知道讀音（按鍵是行、滑動方向是段），測驗效果等價。
    - 正規化：trim、轉小寫、全形英數→半形、片假名→平假名。
    - 判定正確：符合 romaji／alternates，或等於題目假名本身。
    - IME 組字中（`isComposing`）的 Enter 不觸發送出。

- 兩種模式共用成績總表；計時定義不變（顯示題目→送出／點選）。

- 鍵盤平移問題的最終解法（2026-07-21 定案，Chrome 與 Safari 實機驗證通過；純 CSS，不需要 JS 也不需要特殊 meta）：
    - 原則：**讓瀏覽器從頭到尾沒有捲動的理由**。事後修正式的 JS（visualViewport 釘高度、捲動歸零監聽）做過又拆掉——跟瀏覽器搶時序必有肉眼可見的兩段動作。
    - document 永不捲動：body `h-dvh overflow-hidden`，頁面捲動一律發生在 layout 的內層內容區（`overflow-y-auto min-h-0`）。
    - 有輸入框的畫面：輸入框保持在**畫面上半部**，且**任何高度下整欄都不可溢出容器**——假名區用 `shrink`（絕不用 `flex-1` 長大），整欄頂部錨定；鍵盤壓縮 app 時只吃掉輸入框下方的空白，上方內容一個 px 都不動。
    - 反例（勿重蹈）：底部 dock 輸入列（聚焦瞬間必觸發瀏覽器先捲）、輸入框上方放會長大的彈性區（把輸入框推低）。
